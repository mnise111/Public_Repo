## Lab plan

**Topology**

```
Ubuntu source VM (Azure)  ──►  App Gateway (TCP listener)  ──►  Ubuntu backend VM (on-prem)
       (client)                  443, 25, 8490, 8495                  (server)
```

**What each port proves**

| Port | Server behaviour | What it validates |
|---|---|---|
| 443 | TLS echo (self-signed cert) | TLS end-to-end, cert reaches client unchanged |
| 25 | Plain TCP echo | Two independent TCP handshakes, no TLS at all |
| 8490 | Long-running TLS stream (1 msg/s for hours) | Connection pinned to one backend, idle timeout behaviour |
| 8495 | Bulk TLS data transfer (100 MB) | Bytes traverse intact, payload preserved |

All four use **TCP listeners + TCP backend settings** on App Gateway. No certs uploaded to App Gateway anywhere.

---

## Step 1 — Backend VM (on-prem Ubuntu)

SSH in. Everything below runs in `tmux` so it survives logout.

### 1a. Install prerequisites and generate one cert

```bash
sudo apt update && sudo apt install -y tmux openssl ncat python3
mkdir -p ~/lab && cd ~/lab
openssl req -x509 -newkey rsa:2048 -nodes -days 30 \
  -keyout backend.key -out backend.crt \
  -subj "/CN=backend-proves-e2e-tls"
```

### 1b. Open the firewall (adjust to your environment)

```bash
sudo ufw allow 25/tcp
sudo ufw allow 443/tcp
sudo ufw allow 8490/tcp
sudo ufw allow 8495/tcp
```

Also: on-prem firewall must permit inbound from App Gateway's egress IP(s) on those four ports. Get those from the AppGW's outbound public IP or, if private, the AppGW subnet range.

### 1c. Create the four listeners as one tmux session

```bash
tmux new -d -s lab
tmux send-keys -t lab:0 'sudo ncat -lk -p 25 -c "cat"' C-m
tmux split-window -t lab -v
tmux send-keys -t lab:1 'sudo ncat --ssl --ssl-cert ~/lab/backend.crt --ssl-key ~/lab/backend.key -lk -p 443 -c "cat"' C-m
tmux split-window -t lab -h
tmux send-keys -t lab:2 'sudo ncat --ssl --ssl-cert ~/lab/backend.crt --ssl-key ~/lab/backend.key -lk -p 8490 -c "cat"' C-m
tmux select-pane -t lab:0
tmux split-window -t lab -h
tmux send-keys -t lab:3 'sudo ncat --ssl --ssl-cert ~/lab/backend.crt --ssl-key ~/lab/backend.key -lk -p 8495 -c "cat"' C-m
```

What this does: one `tmux` session named `lab` with four panes, each running an `ncat` echo server. `-lk` = listen, keep accepting new connections. `-c "cat"` = pipe everything received back to the sender (echo). Sessions persist after `exit` from SSH.

Verify:
```bash
ss -lntp | grep -E ':25|:443|:8490|:8495'
```
You should see four LISTEN sockets. Re-attach later with `tmux attach -t lab`.

### 1d. Make App Gateway's health probe pass

App Gateway's default TCP probe just needs a successful 3-way handshake on the backend port. `ncat -lk` accepts connections — that's enough. **No probe configuration needed beyond the default.** Health will go green within ~30 seconds of probe start.

If you want to confirm the probe is hitting:
```bash
sudo tcpdump -i any -nn 'tcp[tcpflags] & tcp-syn != 0 and (port 25 or port 443 or port 8490 or port 8495)' &
```
You'll see SYN packets every ~30s from the App Gateway instance IPs.

## Step 2 — App Gateway configuration

Standard_v2, dedicated subnet, frontend IP (public or private — your choice). Then:

| Object | Setting |
|---|---|
| Listener `lst-25` | TCP, port 25 |
| Listener `lst-443` | TCP, port 443 |
| Listener `lst-8490` | TCP, port 8490 |
| Listener `lst-8495` | TCP, port 8495 |
| Backend pool `pool-onprem` | IP/FQDN of the backend VM |
| Backend setting `bs-25` | TCP, port 25, timeout 30s |
| Backend setting `bs-443` | TCP, port 443, timeout 30s |
| Backend setting `bs-8490` | TCP, port 8490, **timeout 3600s** (long-running) |
| Backend setting `bs-8495` | TCP, port 8495, timeout 300s |
| Rules | listener-X → pool-onprem with bs-X (4 rules) |

Confirm **Backend health** in the portal shows all four entries Healthy before continuing.

## Step 4 — Test cases

### Test A — Two TCP handshakes (port 25, plain TCP, no TLS in the way)

**Source VM:**
```bash
sudo tcpdump -i any -nn -s0 -w client-25.pcap 'tcp port 25' &
echo "hello-tcp-only" | ncat $APPGW 25
sleep 2 && sudo pkill tcpdump
```

**Backend VM** (separate SSH or tmux pane):
```bash
sudo tcpdump -i any -nn -s0 -w backend-25.pcap 'tcp port 25' &
# wait for client to send, then
sleep 5 && sudo pkill tcpdump
```

**Compare** (on whichever VM you copy both pcaps to):
```bash
tshark -r client-25.pcap -Y 'tcp.flags.syn==1 && tcp.flags.ack==0' \
  -T fields -e ip.src -e tcp.srcport -e ip.dst -e tcp.dstport -e tcp.seq_raw
tshark -r backend-25.pcap -Y 'tcp.flags.syn==1 && tcp.flags.ack==0' \
  -T fields -e ip.src -e tcp.srcport -e ip.dst -e tcp.dstport -e tcp.seq_raw
```

**Expected:**
- Client pcap: SYN from `client_VM_ip:ephemeral` → `appgw_ip:25`, ISN = `X`
- Backend pcap: SYN from `appgw_instance_ip:ephemeral` → `backend_ip:25`, ISN = `Y`
- `X ≠ Y`, source IPs different, source ports different. **Two handshakes proven.**

### Test B — TLS end-to-end (port 443)

**Both VMs:** start tcpdump on port 443 (same pattern as Test A).

**Source VM:**
```bash
echo | openssl s_client -connect $APPGW:443 -showcerts 2>&1 | tee sclient-443.out
```

Stop captures.

**Verify the cert came from the backend, not App Gateway:**
```bash
grep -A1 'subject=' sclient-443.out
```
Expected: `subject=CN = backend-proves-e2e-tls`. App Gateway has no cert — this CN only exists on the backend.

**Verify TLS handshake bytes are identical on both legs:**
```bash
tshark -r client-443.pcap  -Y 'tls.handshake.type==1' -T fields -e tls.handshake.random
tshark -r backend-443.pcap -Y 'tls.handshake.type==1' -T fields -e tls.handshake.random
```
Expected: identical 32-byte hex value. Same `client_random` on both legs = same TLS session, just shoveled through. **TLS end-to-end proven.**

### Test C — Long-running TLS stream pinned to one backend (port 8490)

This validates that once the frontend TCP connection is established, every byte stays glued to the same backend — including hours later.

**Source VM**, start a session that sends one timestamped line per second:
```bash
tmux new -d -s longrun
tmux send-keys -t longrun \
  "while true; do date -u +%FT%TZ; sleep 1; done | openssl s_client -connect $APPGW:8490 -quiet 2>/dev/null | tee longrun.log" C-m
```

Let it run. Detach SSH. Come back in an hour. Check:
```bash
tail longrun.log     # timestamps still flowing
wc -l longrun.log    # ≈ 3600 lines per hour
```
On the **backend**, while it's running:
```bash
ss -tnp | grep :8490
```
You'll see **one** ESTABLISHED socket from a single App Gateway instance IP. It does not move. **Pinning proven.**

To stop: `tmux kill-session -t longrun`.

### Test D — Bulk data transfer integrity (port 8495)

Validates that 100 MB of data goes through unmodified.

**Backend VM** — verify the cert (already done) and confirm `ncat` on 8495 is up.

**Source VM:**
```bash
# generate 100MB of random data with a known checksum
head -c 100M /dev/urandom > payload.bin
sha256sum payload.bin
# transfer it through AppGW with TLS, capture echoed bytes
ncat --ssl $APPGW 8495 < payload.bin > received.bin &
NCAT_PID=$!
sleep 30   # adjust if your link is slow
kill $NCAT_PID 2>/dev/null
sha256sum received.bin
```

**Expected:** the two SHA-256 sums match. 100 MB of TLS-encrypted bytes traversed two TCP terminations and arrived byte-identical. **Payload integrity proven.**

## Caveats

- App Gateway's TCP idle timeout default is 4 minutes for L4 listeners; if Test C drops, raise the backend setting timeout (already set to 3600s above) and confirm in the portal.
- If your on-prem firewall does deep inspection on port 25, it may reject non-SMTP traffic — `ncat` echo isn't SMTP. Either pick a different plain-TCP port for Test A or whitelist the traffic.
- `ncat --ssl-cert` requires `ncat` from the `ncat` package, not the BusyBox stub. The `apt install ncat` above gets the right one.
