## What the doc actually says

The default draining timeout value for backend servers is 30 seconds. At present, a user-defined draining value is not supported. and A configuration update (PUT) on Application Gateway ends the active connections after the default draining timeout period.

## Theoretical interruption (your topology)

8 instances, 3 backend VMs, long-running TLS-over-TCP:

| Phase | Duration | What happens to your connection |
|---|---|---|
| New config applied | seconds | New connections start landing on reconfigured instances. Existing connection unaffected. |
| Draining window | **up to 30 s** | Your existing TCP/TLS connection keeps flowing on the old instance. |
| Drain expiry | instant | Old instance closes (FIN or RST) your connection. Client must reconnect. |
| Reconnect | 1–3 s | New TCP + new TLS handshake to a (possibly different) AppGW instance and backend VM. |

**Per-connection interruption ≈ time-from-FIN to successful-reconnect ≈ 1–3 s.** It is not gradual or partial. The connection is killed cleanly at the 30-second mark.

With 8 instances, the FIN doesn't arrive at all 8 simultaneously — Azure rolls the change instance-by-instance, so different long-lived connections will be cut at different times across roughly the deployment window (minutes), but each individual connection still sees the same 1-3 s gap.

## Lab — measure it yourself

Reuse the lab from before. Use **port 8490** (long-running TLS).

### Source VM — start a timestamped TLS stream and log every event

One-liner (run in tmux so it survives logout):
```bash
tmux new -d -s drain "while true; do echo CONNECT $(date -u +%s.%N); { while true; do date -u +%s.%N; sleep 1; done; } | openssl s_client -connect $APPGW:8490 -quiet 2>&1; echo DISCONNECT $(date -u +%s.%N); sleep 1; done | tee ~/drain.log"
```

What it does: opens a TLS connection, sends one Unix timestamp per second, logs `CONNECT`/`DISCONNECT` markers around each session, and reconnects automatically when the connection drops.

### Backend VM — confirm sockets churn at change time

```bash
tmux new -d -s sock "while true; do echo --- $(date -u +%s.%N); ss -tn state established '( sport = :8490 )'; sleep 1; done | tee ~/sock.log"
```

### Trigger an Application Gateway configuration change

Pick any benign change that issues a PUT. Examples:
- Add a tag to the Application Gateway resource (Portal → Tags → add → Apply).
- Change Min instance count by 1 in autoscale settings.
- Add a dummy listener on an unused port.

Note the exact UTC time you click **Save**:
```bash
date -u +%s.%N
```

### After the change settles, measure the gap

```bash
awk '/DISCONNECT/{d=$2} /CONNECT/{if(d){printf "gap=%.2fs at %s\n",$2-d,$2; d=0}}' ~/drain.log
```

You will see one (or more, depending on which instance handled your connection) gap entries. Each `gap` value is your **practical interruption per change** for that connection.

### Cross-check against the 30s drain claim

Find when your `DISCONNECT` happened relative to your save click. If save was at `T`, expect `DISCONNECT` somewhere between `T` and `T+30s`. The reconnect (next `CONNECT`) follows immediately after.

## What the numbers will tell you

| Observation | Interpretation |
|---|---|
| Gap < 5 s, single occurrence | Normal — drain-then-reconnect on one instance |
| Gap close to 30 s with no reconnect | Server-side close came at drain expiry; client retried fast |
| Multiple gaps over several minutes | Different AppGW instances rolled at different times during the deployment |
| No gap at all | Your connection happened to live on an instance that wasn't restarted in this PUT (possible but not guaranteed) |

Plan to absorb a **30 s worst-case cut + ~2 s reconnect = ~32 s** application-visible interruption per long-running connection per config change. Build retry/reconnect into the application — you've now demonstrated it works.
