## Lab blueprint

Three things to prove, on four ports, with health probes green and real data flowing.

| Element | Detail |
|---|---|
| Source | Ubuntu VM in Azure (`client-vm`) |
| Proxy | App Gateway Standard_v2, private or public frontend |
| Backend | Ubuntu VM on-prem reachable via VPN/ExpressRoute (`backend-vm`) |
| Ports | 443, 25, 8490, 8495 — all configured as **TCP listener + TCP backend** |
| Cert | One self-signed cert on the backend with CN `backend-e2e-proof`, used for all four ports |

The cert lives **only** on the backend. App Gateway holds no key material — that's what makes the test conclusive.

## Step 1 — Backend VM setup

Install tools and generate one cert reused across all four listeners:

```bash
sudo apt update && sudo apt install -y openssl socat tcpdump tshark netcat-openbsd
mkdir -p ~/lab && cd ~/lab
openssl req -x509 -newkey rsa:2048 -nodes -days 30 \
  -keyout backend.key -out backend.crt \
  -subj "/CN=backend-e2e-proof"
```

Run **four long-running TLS echo servers**, one per port. `socat` is ideal — it keeps connections open, echoes data, logs cleanly, and survives the App Gateway health probe:

```bash
# Each line is one daemon. Run in tmux/screen panes or as systemd units.
sudo socat -v OPENSSL-LISTEN:443,reuseaddr,fork,cert=backend.crt,key=backend.key,verify=0  EXEC:'/bin/cat' &
sudo socat -v OPENSSL-LISTEN:25,reuseaddr,fork,cert=backend.crt,key=backend.key,verify=0   EXEC:'/bin/cat' &
sudo socat -v OPENSSL-LISTEN:8490,reuseaddr,fork,cert=backend.crt,key=backend.key,verify=0 EXEC:'/bin/cat' &
sudo socat -v OPENSSL-LISTEN:8495,reuseaddr,fork,cert=backend.crt,key=backend.key,verify=0 EXEC:'/bin/cat' &
```

Each daemon: TLS-terminates on the backend, echoes any bytes it receives back to the sender. Long-running because `fork` keeps the parent listening forever and `cat` runs until the client closes.

Open the firewall (on-prem) for the App Gateway's outbound IP range to reach 25/443/8490/8495.

## Step 2 — Make the App Gateway health probe go green

This is the part that trips people up. App Gateway's **default TCP probe** sends a SYN, expects SYN-ACK, then RSTs. `socat` accepts the SYN, the kernel sends SYN-ACK, App Gateway RSTs — that's a healthy probe. ✅

But: the OPENSSL listener will then log "TLS handshake failed" once the probe disconnects. That's noise, not a problem. If you want clean logs, use a **custom TCP probe** with these settings on each backend setting:

| Setting | Value |
|---|---|
| Protocol | TCP |
| Port | (same as backend port) |
| Interval | 30s |
| Timeout | 10s |
| Unhealthy threshold | 3 |

The probe is L4 only — it does not attempt TLS. SYN/SYN-ACK is sufficient for "healthy." Verify with **Backend health** in the portal: all four pool members should show **Healthy** within 60 seconds.

If you want belt-and-braces, run this on the backend to confirm the probe is arriving:
```bash
sudo tcpdump -i any -nn 'tcp[tcpflags] & tcp-syn != 0 and (port 443 or port 25 or port 8490 or port 8495)'
```
You should see SYNs from App Gateway's instance IPs every 30s on all four ports.

## Step 3 — App Gateway configuration

```text
Frontend IP: 1 private (or public) IP — call it APPGW_IP
Backend pool: backend-pool → backend-vm IP

Listeners (4):
  L-443  : TCP / 443
  L-25   : TCP / 25
  L-8490 : TCP / 8490
  L-8495 : TCP / 8495

Backend settings (4):
  BS-443  : TCP / 443  / probe=tcp-443
  BS-25   : TCP / 25   / probe=tcp-25
  BS-8490 : TCP / 8490 / probe=tcp-8490
  BS-8495 : TCP / 8495 / probe=tcp-8495

Routing rules (4):
  L-443  → backend-pool via BS-443
  L-25   → backend-pool via BS-25
  L-8490 → backend-pool via BS-8490
  L-8495 → backend-pool via BS-8495
```

No certificate uploaded. No `EnableClientIpPreservation` (turn off so the backend sees AppGW IP — makes the SNAT visible).

## Step 5 — Test cases

### Test A: short-lived TLS handshake on each port (proves end-to-end TLS)

Run on the client:
```bash
for P in 443 25 8490 8495; do
  echo "--- port $P ---"
  echo | openssl s_client -connect APPGW_IP:$P -servername backend-e2e-proof 2>/dev/null \
    | openssl x509 -noout -subject -issuer
done
```
Expected output, four times:
```
subject= CN = backend-e2e-proof
issuer = CN = backend-e2e-proof
```
**This alone proves TLS is end-to-end.** App Gateway has no cert with CN `backend-e2e-proof` — it cannot have manufactured this. The client received the backend's actual cert.

### Test B: long-running TLS connection with bidirectional data (proves data flows through, TCP holds)

Open four shells on the client. In each, run an interactive TLS session and leave it open:

```bash
# shell 1
openssl s_client -connect APPGW_IP:443 -servername backend-e2e-proof -quiet
# now type lines — each line round-trips through the echo server
hello-from-443
hello-from-443     ← echoed back

# shell 2-4: same with 25, 8490, 8495
```
Type periodically over **15+ minutes** to confirm the connection survives. Every echoed line is encrypted client↔backend, opaquely shoveled by App Gateway.

### Test C: bulk data transfer (proves throughput, not just keepalive)

On the backend, replace one of the echo daemons temporarily with a sink+source for measurable transfer:

```bash
# backend: serve a 100 MB file over TLS on 8490
dd if=/dev/urandom of=/tmp/payload.bin bs=1M count=100
sudo socat -v OPENSSL-LISTEN:8490,reuseaddr,fork,cert=backend.crt,key=backend.key,verify=0 \
  SYSTEM:'cat /tmp/payload.bin'
```

On the client:
```bash
openssl s_client -connect APPGW_IP:8490 -servername backend-e2e-proof -quiet 2>/dev/null \
  | sha256sum
```

On the backend, separately:
```bash
sha256sum /tmp/payload.bin
```

Hashes match → 100 MB of TLS-encrypted data flowed end-to-end through App Gateway intact. Restore the echo daemon afterward.

### Test D: dual capture — the empirical proof

The decisive test. Capture simultaneously on both ends.

**Backend** (in one shell):
```bash
sudo tcpdump -i any -nn -s0 -w /tmp/backend.pcap 'tcp port 443 or tcp port 25 or tcp port 8490 or tcp port 8495'
```

**Client** (in another shell):
```bash
sudo tcpdump -i any -nn -s0 -w /tmp/client.pcap 'tcp port 443 or tcp port 25 or tcp port 8490 or tcp port 8495'
```

**Client** — trigger one connection per port:
```bash
for P in 443 25 8490 8495; do echo | openssl s_client -connect APPGW_IP:$P 2>/dev/null >/dev/null; done
```

Stop both captures.

Now run these four queries — they are the entire proof:

**(1) Two TCP handshakes per session, with different IPs:**
```bash
echo "=== Client side: who client talked to ==="
tshark -r /tmp/client.pcap -Y 'tcp.flags.syn==1 and tcp.flags.ack==0' \
  -T fields -e ip.src -e ip.dst -e tcp.dstport -e tcp.seq_raw
echo "=== Backend side: who connected to backend ==="
tshark -r /tmp/backend.pcap -Y 'tcp.flags.syn==1 and tcp.flags.ack==0' \
  -T fields -e ip.src -e ip.dst -e tcp.dstport -e tcp.seq_raw
```
Client pcap shows: `clientVM → APPGW_IP`. Backend pcap shows: `APPGW_instance_IP → backendVM`. Different src IPs. Different initial sequence numbers. → **TCP terminated twice, confirmed.**

**(2) Same TLS ClientHello on both legs (the smoking gun for end-to-end TLS):**
```bash
tshark -r /tmp/client.pcap  -Y 'tls.handshake.type==1' -T fields -e tcp.dstport -e tls.handshake.random > /tmp/c_random.txt
tshark -r /tmp/backend.pcap -Y 'tls.handshake.type==1' -T fields -e tcp.dstport -e tls.handshake.random > /tmp/b_random.txt
diff <(sort /tmp/c_random.txt) <(sort /tmp/b_random.txt)
```
No diff output means the 32-byte `client_random` value the client generated appears **byte-identical** in the backend's pcap — for all four ports. App Gateway forwarded the handshake; it did not re-originate one. → **TLS terminated once, confirmed.**

**(3) Backend's cert reached the client unmodified:**
```bash
tshark -r /tmp/client.pcap -Y 'tls.handshake.type==11' -V 2>/dev/null \
  | grep -E 'commonName|RDNSequence' | head -20
```
You'll see `commonName=backend-e2e-proof`. App Gateway has no such cert.

**(4) Destination port preserved, source port rewritten:**
```bash
tshark -r /tmp/backend.pcap -Y 'tcp.flags.syn==1 and tcp.flags.ack==0' \
  -T fields -e tcp.srcport -e tcp.dstport
```
Source ports = high ephemeral values from App Gateway (different per connection). Destination ports = 25/443/8490/8495 exactly. → SNAT on src, DNAT preserves dst.

## Step 6 — Result interpretation cheat sheet

| Observation | Conclusion |
|---|---|
| Two SYNs per session, different src IPs | TCP terminated twice |
| `tls.handshake.random` matches between client.pcap and backend.pcap | TLS handshake passed through, not re-originated |
| Cert subject = `backend-e2e-proof` on the client | TLS terminated at backend |
| `s_client` works without uploading any cert to AppGW | AppGW held no key, yet TLS succeeded → it's not in the TLS path |
| 100 MB hash matches | Encrypted payload traversed AppGW unmodified |
| All four ports show identical behavior | Result is general, not port-specific |

If all six rows hold, you've validated the claim on real wire data, on four ports, with a long-lived connection and a real bulk transfer. The backend's cert and the matching `client_random` are the two pieces of evidence that no amount of marketing copy can refute — they're cryptographic proof of where TLS actually terminated.
