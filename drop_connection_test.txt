## What the docs actually say

A configuration update (PUT) on Application Gateway ends the active connections after the default draining timeout period. The default draining timeout is 30 seconds, and a user-defined draining value is not supported.

## Theoretical interruption per config change

With 8 instances, App Gateway updates them in batches (rolling). For a long-running TLS-over-TCP connection pinned to one instance:

| Event on your instance | Expected impact on your connection |
|---|---|
| Instance not in the current batch | 0 ms — connection unaffected |
| Instance in the batch being updated | Connection enters draining; **closed at ~30s** if still open |
| Reconnect after drop | New TCP + TLS handshake against a healthy instance, typically **<1s** |

So per config change, **either no interruption at all, or one ~30s drain followed by a ~1s reconnect** — depending on which of the 8 instances your flow happens to be pinned to.

## Minimal lab — one client, one server, one timeline

Use your existing port 8490 setup. You only need to add a "heartbeat with timestamp" so you can see the exact second the connection drops and the exact second the reconnect succeeds.

### Backend (already running from previous lab)

`ncat --ssl ... -lk -p 8490 -c cat` — nothing to change.

### Source VM — one line that logs every event with a timestamp

```bash
while true; do echo "$(date -u +%FT%T.%3NZ) RECONNECT"; (while true; do date -u +%FT%T.%3NZ; sleep 1; done) | openssl s_client -connect $APPGW:8490 -quiet 2>/dev/null; done | tee -a heartbeat.log
```

What this does: opens a TLS connection through App Gateway, sends one timestamp per second, echoes received timestamps to `heartbeat.log`, and **the moment the connection drops**, the inner loop exits and the outer `while` immediately reconnects, logging `RECONNECT` with a timestamp. Run inside `tmux` so you can leave it:

```bash
tmux new -d -s hb 'while true; do echo "$(date -u +%FT%T.%3NZ) RECONNECT"; (while true; do date -u +%FT%T.%3NZ; sleep 1; done) | openssl s_client -connect $APPGW:8490 -quiet 2>/dev/null; done | tee -a heartbeat.log'
```

### Trigger a config change

In the Azure portal, do something harmless that forces a PUT — easiest is **add a tag** to the App Gateway resource and Save, or change min instance count by 1 and revert. Note the wall-clock time you click Save.

### Read the result

```bash
grep -B1 RECONNECT heartbeat.log | tail -20
```

You'll see a pattern like:
```
2026-05-01T10:14:32.104Z      <- last timestamp echoed before drop
2026-05-01T10:15:02.881Z RECONNECT   <- reconnect succeeded
```

The gap between those two timestamps **is your interruption duration** for this connection on this config change.

## What to expect across multiple changes

Run 5–10 config changes (with a few minutes between each) and tabulate the gaps:

| Change # | Gap (s) | Interpretation |
|---|---|---|
| 1 | ~0 | your instance wasn't in the batch |
| 2 | ~31 | your instance was in the batch — drained at 30s + ~1s reconnect |
| 3 | ~0 | not in batch |
| ... | | |

With 8 instances, statistically your single connection lands in the affected batch on a fraction of changes (depends on Azure's batch size — typically 1–2 instances at a time). Most changes are invisible to you; the ones that hit your instance show ~30s.

## Bottom line for your design

Worst-case per config change for any one long-running flow: **≈30 seconds**, deterministic, then auto-recovers. To make the application resilient, the client must **reconnect on TCP close** — exactly what the one-liner above does. If your real client doesn't auto-reconnect, that's the gap to close in the application, not in App Gateway.
