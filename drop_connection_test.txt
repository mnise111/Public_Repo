## Theoretical behaviour

From the docs: The default draining timeout value for backend servers is 30 seconds. At present, a user-defined draining value is not supported. And: A configuration update (PUT) on Application Gateway ends the active connections after the default draining timeout period.

So per config change: **up to 30 seconds** of in-flight connections being drained, then forcibly closed. Existing connections on instances *not* being recycled may survive; connections on recycled instances are killed. With 8 instances and a rolling update, you should expect **a portion** of long-lived TCP connections to drop per change, with each affected client reconnecting within ~30s of the drain.

## What you'll measure

Time between **last byte received** on the old connection and **first byte received** on the new connection. That's your real interruption.

## Lab setup

### Backend (one tmux pane is enough ‚Äî reuse port 8490 from your existing lab)

```bash
tmux new -d -s recon ; tmux send-keys -t recon "while true; do date -u +%FT%T.%3NZ; sleep 0.1; done | sudo ncat --ssl --ssl-cert ~/lab/backend.crt --ssl-key ~/lab/backend.key -lk -p 8490" C-m
```

This streams a UTC timestamp every 100ms to whoever connects. One line ‚âà 100ms granularity for measuring gaps.

### Source VM (Azure Ubuntu) ‚Äî single resilient client

Open 8 parallel long-lived clients so you can see how many of the 8 AppGW instances are affected:

```bash
for i in 1 2 3 4 5 6 7 8; do tmux new -d -s c$i "while true; do echo \"--- RECONNECT $(date -u +%FT%T.%3NZ) ---\"; ncat --ssl $APPGW 8490 < /dev/null; sleep 0.05; done | tee ~/lab/c$i.log" ; done
```

Each session loops: connect, drain whatever the server sends until the socket closes, log a `--- RECONNECT <timestamp> ---` marker, reconnect. Logs land in `~/lab/c1.log` ‚Ä¶ `c8.log`.

Let them run for ~60s to confirm baseline (no reconnect markers should appear).

## Trigger a config change

Pick a no-op-ish change that still issues a PUT. From Azure Cloud Shell or `az` CLI:

```bash
az network application-gateway update -g <RG> -n <APPGW> --set tags.test=$(date +%s)
```

A tag change forces a control-plane PUT and triggers the drain/reconfigure cycle without altering data-plane behaviour. Note the start time.

## Measure interruption

After the update completes (the CLI returns), inspect each client log:

```bash
for i in 1 2 3 4 5 6 7 8; do echo "=== client $i ==="; grep -B1 -A1 RECONNECT ~/lab/c$i.log | tail -12; done
```

You'll see something like:
```
2026-05-01T04:12:18.314Z      ‚Üê last timestamp from old connection
--- RECONNECT 2026-05-01T04:12:46.902Z ---
2026-05-01T04:12:47.005Z      ‚Üê first timestamp from new connection
```

**Gap = `RECONNECT timestamp` ‚àí `last timestamp before it`.** That's your interruption window for that client.

Compute all 8 gaps in one shot:

```bash
for i in 1 2 3 4 5 6 7 8; do awk '/RECONNECT/{split($0,a," "); rec=a[3]; gsub("Z","",rec); gsub("Z","",prev); cmd="date -d "prev" +%s.%3N"; cmd|getline t1; close(cmd); cmd="date -d "rec" +%s.%3N"; cmd|getline t2; close(cmd); printf "client '$i' gap = %.3fs\n", t2-t1; next} {prev=$0}' ~/lab/c$i.log; done
```

## Repeat with different change types

Run several updates and tabulate. Examples to try:

| Change | Command |
|---|---|
| Tag-only PUT | `az network application-gateway update -g RG -n GW --set tags.t=$(date +%s)` |
| Add HTTP listener | `az network application-gateway http-listener create ...` |
| Modify backend setting timeout | `az network application-gateway http-settings update ... --timeout 35` |
| Add a probe | `az network application-gateway probe create ...` |
| Capacity change (8‚Üí9 instances) | `az network application-gateway update ... --set autoscaleConfiguration.minCapacity=9` |

## Expected pattern

- **Tag-only PUT:** often **0 drops** on most/all clients ‚Äî control-plane only.
- **Real config change (listener/backend/probe):** rolling instance recycle. Typically **1‚ÄìN out of 8 clients** drop, with gaps **between a few seconds and ~30s** each, matching the drain window.
- **Capacity change:** new instances spin up; existing connections usually unaffected, but verify.

Cleanup:
```bash
tmux kill-session -t recon ; for i in 1 2 3 4 5 6 7 8; do tmux kill-session -t c$i; done
```

Result: you'll have empirical, per-client, sub-second-accurate measurements of the actual interruption your long-lived TLS-over-TCP connections experience for each type of change in your environment.
